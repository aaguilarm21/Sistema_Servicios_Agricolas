DESARROLLO DEL SOFTWARE - Sistema de Servicios Agrícolas

Este documento describe el desarrollo del proyecto paso a paso, con el objetivo de que otras personas entiendan de qué trata el sistema, cómo se construyó y cuál es su propósito.

1. Propósito del proyecto
-------------------------
El proyecto es un sistema web de gestión para servicios agrícolas. Está diseñado para registrar y administrar información sobre usuarios, catálogos de proveedores y maquinaria, operaciones agrícolas, y reportes. El objetivo es facilitar el control de los procesos operativos y administrativos en un ambiente agrícola.

2. Tecnologías utilizadas
--------------------------
- Framework: Django 6.x
- Base de datos: SQLite (archivo `BaseDatos/db.sqlite3`)
- Lenguajes: Python para backend, HTML/CSS/JavaScript para frontend
- Estructura de plantillas: Django Templates en `Frontend/templates`
- Estilos y scripts: Archivos estáticos en `Frontend/static`

3. Estructura del proyecto
--------------------------
- `miproyecto/`: configuración central de Django, rutas principales y ajustes.
- `accounts/`: aplicación local dedicada a la gestión de usuarios, autenticación, modelos y APIs.
- `Backend/`: contiene las vistas que renderizan las páginas y validan permisos.
- `Frontend/`: aloja las plantillas HTML y los estilos del sitio.
- `BaseDatos/`: almacena la base de datos SQLite del proyecto.

4. Desarrollo de la funcionalidad
---------------------------------
4.1. Autenticación y roles
- Se implementó un login personalizado (`CustomLoginView`) y una vista AJAX para iniciar sesión desde un modal.
- Se utiliza el sistema de usuarios de Django y se amplió con el modelo `UserProfile` para agregar código y puesto.
- El proyecto diferencia dos tipos de perfiles: administradores y usuarios normales, mediante `is_superuser` y grupos de Django.

4.2. Módulos del sistema
- El menú principal se encuentra en `/modulos/` y ofrece acceso a varios módulos:
  - Usuarios
  - Configuración y catálogos
  - Registros operativos
  - Operación agrícola / logística
  - Reportes
  - Base de datos de registros (solo administradores)
- El acceso a cada ruta se controla desde el backend, restringiendo módulos sensibles a usuarios con rol administrador.

4.3. Catálogos y datos maestros
- Se crearon modelos para catálogos relevantes al negocio agrícola:
  - Proveedor, Empleado, Maquinaria, Bodega, Artículo
  - Labor, Cuenta, UnidadMedida, Variedad, TipoMaquina, Marca, Municipio, Auxiliar
- Estos modelos facilitan la administración de datos referenciales que se usan en el sistema.

4.4. Registros operativos persistentes
- El formulario de `registros_operativos` guarda datos en el modelo `RegistroOperativo`.
- Se almacenan campos como boleta, fecha, tipo de servicio, proveedor, finca, lote, maquinaria, operador, costos y observaciones.
- El registro es visible en una tabla en `/registros-data/`, accesible solo a administradores.

4.5. UI y experiencia de usuario
- Se creó una interfaz de módulos con botones limpios y un modal para cambiar de usuario.
- Los mensajes de advertencia y errores aparecen en un popup estilizado.
- Se separó el diseño (CSS) de la plantilla HTML para mantener la lógica y el estilo independientes.

5. Implementación técnica
-------------------------
5.1. Rutas principales
- `miproyecto/urls.py` define las rutas del sitio incluyendo:
  - `/` -> `home`
  - `/modulos/` -> panel de módulos
  - `/configuracion/` -> catálogos administrativos
  - `/registros/` -> formulario de registros
  - `/registros-data/` -> listado de registros operativos

5.2. Backend y lógica de vistas
- `Backend/views.py` centraliza la lógica de negocio y los controles de acceso.
- Cada vista verifica permisos antes de renderizar o procesar datos.
- El módulo de usuarios incluye creación segura de usuarios desde el backend con JSON.

5.3. API internas
- `accounts/views.py` expone APIs internas:
  - `ajax_login` para login por AJAX
  - `buscar_usuario_por_codigo` para buscar empleados por código
  - CRUD de proveedores y empleados mediante JSON

5.4. Modelos y migraciones
- `accounts/models.py` define los modelos de datos.
- El proyecto utiliza migraciones de Django para crear y actualizar las tablas en SQLite.

6. Cómo se realizó el desarrollo
--------------------------------
- El proyecto partió de una base Django estándar con una app local `accounts`.
- Se agregaron funcionalidades específicas del dominio agrícola, como registros operativos y catálogos.
- Se trabajó en la seguridad de acceso usando roles y permisos Django.
- Se construyó la interfaz en `Frontend/templates` y se añadió interacción con JavaScript para el modal de cambio de usuario.
- Se mejoró la separación de responsabilidades moviendo estilos CSS fuera de las plantillas.

7. Resultado final
------------------
El sistema resultante es una aplicación web de administración para servicios agrícolas, con:
- login y control de sesiones
- roles de usuario y administrador
- módulos de gestión y reportes
- almacenamiento de registros operativos
- acceso condicional a datos sensibles
- interfaz simple con estilos coherentes

8. Recomendaciones futuras
--------------------------
- Migrar a una base de datos más robusta para producción (PostgreSQL, MySQL).
- Añadir validaciones de formulario en el frontend y backend.
- Implementar paginación y filtros en el listado de registros.
- Agregar pruebas automáticas para rutas y modelos.
- Usar un sistema de permisos más detallado para controlar cada acción.
